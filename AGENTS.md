# Правила роботи з проєктом Intelekt Web App

Цей файл описує правила для Codex та інших агентів, які працюють з проєктом.

## Мета проєкту

Проєкт створюється як веб-застосунок для обліку майна з майбутнім розгортанням на Firebase.

Поточна основа:

- локальний статичний застосунок у папці `public`;
- Google-авторизація через Firebase Authentication;
- ролі `admin` і `user`;
- таблиця обліку майна з рядками, колонками, статусами та підрахунками;
- workspace-інтерфейс із таблицями в боковій панелі, блоками таблиць, полями, рядками та підрахунками;
- сторінка налаштувань тільки для керування ролями;
- Firebase Hosting і Firestore rules.

## Дозволені дії без додаткового уточнення

Codex може самостійно:

- читати всі файли в межах цього проєкту;
- редагувати файли в межах цього проєкту;
- створювати нові файли документації, стилів, скриптів і конфігурації;
- виправляти помилки інтерфейсу, логіки таблиць, ролей і локального запуску;
- покращувати структуру `public/index.html`, `public/styles.css`, `public/app.js`;
- оновлювати `README.md`, `AGENTS.md`, `firebase.json`, `firestore.rules`;
- запускати локальні перевірки, якщо вони не змінюють зовнішні сервіси;
- додавати демо-дані, якщо вони не містять приватної інформації.

## Дії, для яких потрібне підтвердження користувача

Перед виконанням потрібно окремо спитати дозвіл:

- видалення великих частин функціоналу або повна заміна архітектури;
- зміна реального Firebase project id;
- деплой у Firebase;
- зміна production rules, якщо це може заблокувати користувачів;
- встановлення нових пакетів через інтернет;
- запис або редагування файлів поза папкою проєкту;
- робота з приватними ключами, токенами, сервісними акаунтами;
- коміти, пуші, reset, rebase або інші git-операції, що змінюють історію.

## Заборонено

Codex не повинен:

- зберігати секретні ключі у відкритому коді;
- додавати Firebase service account json у репозиторій;
- відкривати доступ до Firestore для всіх без авторизації;
- змінювати роль користувача на `admin` з клієнтського коду без перевірки правил;
- видаляти дані користувача без прямого прохання;
- відкатувати зміни користувача без прямого прохання.

## Firebase доступи

Authentication:

- основний спосіб входу: Google;
- неавторизований користувач не має доступу до робочої області;
- перший реальний адміністратор призначається вручну у Firestore Console.

Firestore:

- `users/{uid}` зберігає профіль і роль користувача;
- новий користувач може створити тільки власний профіль з роллю `user`;
- тільки `admin` може змінювати ролі;
- авторизовані користувачі можуть читати таблиці;
- тільки `admin` може змінювати структуру таблиць і дані майна.

Hosting:

- публічна папка: `public`;
- усі маршрути ведуть на `public/index.html`;
- деплой тільки після підтвердження користувача.

## Ролі в застосунку

`admin`:

- бачить таблицю майна;
- створює таблиці;
- перемикається між таблицями з бокової панелі;
- додає блоки таблиць на сторінку;
- налаштовує статус сортування блоку, щоб рядки автоматично переходили між блоками за значенням поля `Статус`;
- додає рядки;
- додає колонки;
- редагує назви та типи полів;
- редагує клітинки;
- змінює статуси;
- видаляє рядки;
- видаляє поля;
- змінює ролі користувачів.

`user`:

- входить через Google або demo mode;
- бачить таблицю майна;
- не може змінювати дані, структуру таблиці або ролі.

## Локальний запуск

Основна команда:

```powershell
powershell -ExecutionPolicy Bypass -File .\start-local.ps1
```

Після запуску сторінка доступна за адресою:

```text
http://localhost:5173
```

Якщо Firebase config ще не заповнений, застосунок працює в demo mode.

## Робота з конфігурацією

Файл `public/firebase-config.js` може містити Firebase web config.

Не додавати сюди:

- service account key;
- private key;
- admin SDK credentials;
- токени доступу;
- паролі.

Web config Firebase не є секретним ключем, але його треба використовувати разом із правильними Firestore rules.

## Стиль розробки

- Перевага простим рішенням без зайвих залежностей.
- Інтерфейс має залишатися українською мовою.
- Для ролей і доступів спочатку оновлювати правила Firestore, потім клієнтський інтерфейс.
- Перед завершенням роботи перевіряти синтаксис JavaScript.
- Не ламати demo mode, бо він потрібен для швидкої перевірки без Firebase.
- Якщо додається нова сутність даних, описати її у цьому файлі або README.

## Поточна модель даних

`users/{uid}`:

```json
{
  "name": "User name",
  "email": "user@example.com",
  "photoURL": "https://...",
  "role": "user"
}
```

`workspaces/default/sheets/main-assets` зберігає workspace:

```json
{
  "name": "Intelekt Workspace",
  "activePageId": "assets",
  "settings": {
    "statusOptions": ["На складі", "В роботі", "Ремонт", "Списано"]
  },
  "pages": [
    {
      "id": "assets",
      "name": "Облік майна",
      "blocks": [
        {
          "id": "main-assets",
          "title": "Основна таблиця",
          "columns": [],
          "rows": []
        }
      ]
    }
  ]
}
```

## Пріоритети наступної розробки

1. Підключити реальний Firebase config.
2. Додати стабільне збереження таблиць у Firestore.
3. Додати кілька таблиць або реєстрів майна.
4. Додати типи колонок: формула, валюта, дата, відповідальний, файл.
5. Додати історію змін.
6. Додати імпорт/експорт CSV або Excel.
7. Додати права на рівні окремих таблиць або підрозділів.
